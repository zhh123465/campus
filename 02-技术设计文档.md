> **项目代号**：CampusForum

## 1. 设计总览

### 1.1 设计原则

| 原则 | 说明 |
|---|---|
| **轻量优先** | 只引入必要中间件，部署门槛低，单台 4C8G 主机即可承载千人在线 |
| **前后端分离** | 前端纯静态发布；后端 RESTful 提供能力 |
| **模块化分层** | 业务按领域分模块，横向能力（搜索、AI、存储）以接口抽象 |
| **多租户友好** | 应用层"租户感知"为一等公民，单实例多租户与独立私有化无缝切换 |
| **开源可二开** | 项目结构标准化、文档齐全、配置外置、扩展点清晰 |
| **演进式架构** | 先单体（Modular Monolith），具备拆分微服务的清晰边界 |

| 层 | 候选 | 选定 | 选定理由 |

### 1.3 架构演进策略

```
阶段一（开发期）：模块化单体（Modular Monolith）
  - 单 Spring Boot 应用，按 package 分模块
  - 模块间通过 Spring 内部 API 调用，禁止反向依赖
  - 每个模块拥有独立的 service / mapper / dto / domain

阶段二（演进期，未来）：可选微服务
  - 按"用户 / 内容 / 互动 / 资源 / 通知 / AI"拆分
  - 模块间已是接口调用，可平滑替换为 Feign/gRPC
```

## 3. 多租户与数据隔离设计（核心研究点）

### 3.1 三种多租户隔离模式对比

| 模式 | 数据库 | 表 | 租户字段 | 隔离强度 | 资源成本 | 适用场景 |
|---|---|---|---|---|---|---|
| 独立数据库 | 一租户一库 | 独立 | 无需 | ★★★★★ | 高 | 大客户独立合规需求 |
| 独立 Schema | 共享库不同 Schema | 独立 | 无需 | ★★★★ | 中 | 中等规模、PG 友好 |
| 共享表 + tenant_id | 共享 | 共享 | 必需 | ★★★ | 低 | 中小规模 SaaS |

### 3.2 本平台采用的混合策略

为兼顾"独立私有化部署"（一所学校一套）与"未来 SaaS 多租户托管"（一套服务多所学校），本平台采用**双模可切换设计**：

- **模式 A：独立私有化（默认）**——每所学校部署一套，`tenant_id` 不参与查询，运行如同单租户系统。
- **模式 B：单实例多租户（SaaS 托管）**——所有业务表带 `tenant_id`，由租户拦截器自动注入查询条件。

切换通过 `application.yml` 中 `tenant.mode = standalone | multi` 一键切换。

### 3.3 实现机制

#### 3.3.1 租户识别（Tenant Resolver）

```
请求进入 → TenantInterceptor 拦截 → 按以下优先级解析 tenant_id：
  1. HTTP Header: X-Tenant-Id（管理员场景）
  2. 子域名：xxx.campusforum.org → tenant_code = xxx
  3. JWT 中的 tenant claim（用户登录后由签发时注入）
  4. 默认 tenant（standalone 模式 = 1）
→ 写入 TenantContext（基于 ThreadLocal + InheritableThreadLocal）
→ 业务结束清理上下文
```

#### 3.3.2 数据隔离（MyBatis-Plus 插件）

利用 MyBatis-Plus 的 `TenantLineInnerInterceptor`，对所有业务表 SQL 自动改写：

```java
// 注册插件
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor i = new MybatisPlusInterceptor();
    i.addInnerInterceptor(new TenantLineInnerInterceptor(new TenantLineHandler() {
        @Override public Expression getTenantId() { 
            return new LongValue(TenantContext.getId()); 
        }
        @Override public boolean ignoreTable(String table) {
            return TenantTables.isIgnored(table);  // 字典表/系统表跳过
        }
    }));
    return i;
}
```

效果：业务代码完全无感，`select * from post where status = 1` 自动改写为 `select * from post where status = 1 and tenant_id = ?`。

#### 3.3.3 文件存储隔离

MinIO 桶规划：

```
campusforum/
├─ tenant-1/                # 租户 1 的所有文件
│   ├─ avatar/
│   ├─ post/
│   ├─ resource/
│   └─ checkin/
├─ tenant-2/
└─ public/                  # 跨租户共享（如平台默认头像）
```

`StorageService.upload()` 内部自动拼接 `tenant-{id}/` 前缀，外部无感。

#### 3.3.4 缓存隔离

Redis Key 命名约定：`{appName}:t{tenantId}:{module}:{key}`，例如：

```
campusforum:t1:user:profile:101
campusforum:t1:post:hot:list
campusforum:t2:user:profile:101    # 不同租户互不干扰
```

封装 `TenantAwareRedisTemplate` 统一处理。

#### 3.3.5 搜索引擎隔离

MeiliSearch 索引按租户分片：`post-t1`、`resource-t1`、`post-t2`，搜索服务层根据 `TenantContext` 选择索引。

#### 3.3.6 跨租户访问防护（双保险）

- **第一保险**：MyBatis 插件自动注入 `tenant_id` 条件
- **第二保险**：Service 层在关键查询返回后，做 `tenant_id` 比对断言；若发现错误数据，立即抛出 `TenantViolationException` 并触发审计告警

### 3.4 隔离测试矩阵

| 测试项 | 测试方法 | 通过标准 |
|---|---|---|
| 用户跨租户登录尝试 | 用 A 租户账号在 B 租户域名登录 | 拒绝登录，记录审计 |
| 用户跨租户查询 | 构造带 B 租户 ID 的查询请求 | 返回空，记录审计告警 |
| 文件直链跨租户访问 | 用 A 租户 token 请求 B 租户文件 URL | 403 |
| 搜索跨租户 | A 租户搜索 B 租户内容 | 命中 0 |
| 缓存逃逸 | 高并发跨租户混合请求 | 缓存零交叉污染 |
| 审计完整性 | 所有跨租户访问尝试 | 100% 进入审计日志 |

## 4. 数据库设计

### 4.1 设计规范

- 主键统一 `bigint unsigned auto_increment`，对外用雪花 ID（`snowflake_id` 字段）
- 所有业务表带：`tenant_id`、`created_at`、`updated_at`、`deleted`（软删除）、`creator_id`
- 字段类型：状态用 `tinyint`、金额/积分用 `bigint`、文本短用 `varchar`、长用 `text`
- 字符集 `utf8mb4` + `utf8mb4_0900_ai_ci`
- 命名：表 `下划线小写复数`，字段 `下划线小写`，索引 `idx_xxx` / `uk_xxx`
- 必备索引：`tenant_id`、外键字段、查询条件常用字段

### 4.4 索引与性能策略

- 列表类查询（feed、空间帖子流）使用 **覆盖索引** + **游标分页**（`id < lastId limit n`），避免深翻页 `OFFSET` 卡顿
- 热点统计（点赞数、评论数）通过冗余字段 + 异步事件刷新，减少 `count(*)` 实时聚合
- 帖子全文搜索：MySQL FULLTEXT（小规模）+ MeiliSearch（推荐）双链路，前期 MySQL 兜底
- 软删除字段加索引以避免全表扫描
- 高频字典数据（学院、专业、敏感词）走 Caffeine 本地缓存

## 5. 关键 API 设计

### 5.1 设计规范

- 风格：RESTful + JSON
- URL 前缀：`/api/v1/`
- 鉴权：Header `Authorization: Bearer <token>`（Sa-Token 签发）
- 多租户：Header `X-Tenant-Id`（仅 SaaS 模式）或子域名解析
- 统一响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": { ... },
  "traceId": "5f3c1b..."
}
```

- 错误码规范：`0` 成功；`1xxxx` 业务错误；`4xxxx` 客户端错误；`5xxxx` 服务端错误
- 分页：`?page=1&size=20` 或 `?cursor=xxx&size=20`
- 时间字段：ISO 8601（`2026-05-01T12:00:00+08:00`）

## 6. 权限模型设计

### 6.1 角色定义

| 域 | 角色 | 编码 |
|---|---|---|
| 平台 | 超级管理员（仅 SaaS） | `SUPER_ADMIN` |
| 校级 | 校级管理员 | `TENANT_ADMIN` |
| 校级 | 普通用户 | `USER` |
| 空间 | 所有者 | `SPACE_OWNER` |
| 空间 | 管理员 | `SPACE_ADMIN` |
| 空间 | 成员 | `SPACE_MEMBER` |

### 6.2 权限粒度

- 全局权限：以 `domain:resource:action` 命名，例如 `tenant:user:ban`、`space:post:delete`
- 资源域权限：通过运行时校验目标资源的 `tenant_id` / `space_id` 与当前用户角色匹配

### 6.3 拦截方案

```java
@SaCheckPermission("space:post:delete")
@PostMapping("/spaces/{id}/posts/{pid}/delete")
public R<?> delete(@PathVariable Long id, @PathVariable Long pid) {
    // Sa-Token 校验全局权限
    // 业务层再校验：当前用户在 spaceId=id 中是否为 OWNER/ADMIN
    spacePermissionService.assertSpaceAdmin(id);
    postService.deleteInSpace(pid, id);
    return R.ok();
}
```

## 7. AI 模块设计

### 7.1 设计目标

- **解耦**：AI 服务以接口形式存在，可一键替换提供商
- **可控**：所有调用通过统一网关，限流、配额、审计完整
- **降级**：AI 服务不可用时，主功能不受影响（摘要生成失败仅影响摘要按钮）

### 7.2 抽象设计

```java
public interface AiService {
    String summarize(String content);
    RiskResult moderate(String content);
    List<String> recommendTags(String title, String content);
    String chat(List<Message> messages, RagContext ragContext);
}

@Service
@ConditionalOnProperty(name = "ai.provider", havingValue = "openai")
public class OpenAiCompatService implements AiService { /* 默认实现 */ }

@Service
@ConditionalOnProperty(name = "ai.provider", havingValue = "ollama")
public class OllamaService implements AiService { /* 本地模型实现 */ }
```

### 7.3 OpenAI 兼容协议

通过 `LangChain4j` 调用，配置中只需修改 `base-url` 与 `api-key`：

```yaml
ai:
  provider: openai          # openai | ollama | mock
  base-url: https://api.deepseek.com/v1   # 也可为本地 http://ollama:11434/v1
  api-key: ${AI_API_KEY:}
  model: deepseek-chat      # 或 qwen2.5-7b-instruct（本地）
  rate-limit:
    per-user-per-min: 5
    per-tenant-per-day: 1000
```

### 7.4 RAG（智能问答助手）

- 文档源：空间内"已采纳问答" + "精华帖" + "公开资源描述"
- 嵌入模型：bge-small-zh（本地）或第三方 embeddings API
- 向量库：MeiliSearch（最新版支持向量检索）/ 简化版用 PostgreSQL pgvector（未来可换）
- 检索策略：BM25（关键词）+ 向量召回 → top-K 重排 → 拼接到 system prompt

### 7.5 内容安全审核流程

```
发帖请求 → 入库前 AI 审核 → 风险等级
  ├─ 0 正常：直接发布
  ├─ 1 中风险：发布但加入审核队列
  └─ 2 高风险：拦截 + 通知用户 + 通知管理员
              + 命中规则记入 sensitive_words 候选库
```

## 10. 安全设计

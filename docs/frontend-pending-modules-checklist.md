# CampusForum 前端待完善模块清单

## 1. 文档目的

本文用于从工程落地角度梳理当前前端“哪些模块已经较成熟、哪些还存在缺口、优先应该补什么”，便于后续：

- 排开发计划
- 拆迭代任务
- 做联调前检查
- 判断哪些页面仍属于视觉稿/半成品状态

本文重点回答四个问题：

1. 哪些页面当前是完整闭环
2. 哪些页面已经有 UI 但交互不完整
3. 哪些地方已经有 API 但前端没接上
4. 哪些问题最值得优先修

---

## 2. 总体结论

当前前端成熟度可以分成三档：

### A. 已形成主闭环

这些页面已经具备“可进入、可加载、可操作、可反馈、可返回”的完整链路：

- 登录 `Login.vue`
- 注册 `Register.vue`
- 忘记密码 `ForgotPassword.vue`
- 广场 `Square.vue`
- 发帖 `PostCreate.vue`
- 帖子详情 `PostDetail.vue`
- 用户主页 `UserPage.vue`
- 关注粉丝 `FollowsList.vue`
- 打卡列表/创建/详情
- 资源列表/上传/详情
- 搜索页主体
- 通知中心
- 消息中心（基础版）
- 后台用户/帖子/空间/举报/敏感词/租户/AI 配置

### B. 已有较完整 UI，但业务闭环不完整

- 首页 `Home.vue`
- 个人中心 `Profile.vue`
- 空间详情 `SpaceDetail.vue`
- AI 助手 `AiAssistant.vue`
- 后台仪表盘 `AdminDashboard.vue`

### C. 结构性缺口/联动断点明显

- 搜索页未消费 query 参数
- 空间详情与当前路由体系不一致
- 消息实时性不足
- API 封装还不完全统一
- 个别页面类型定义与实际接口能力不一致

---

## 3. 按模块拆分的待完善项

## 3.1 首页 `Home.vue`

### 当前状态

- 视觉完成度高
- 适合作为产品展示页
- 但核心数据和内容仍是静态 mock

### 已有能力

- 跳登录/注册/个人中心/广场/空间列表
- 根据登录态切换 CTA

### 待完善项

#### 高优先级

1. 接入首页真实统计数据
   - 已入驻高校数
   - 用户数
   - 帖子数
   - 部落数
   - 今日打卡数

2. 接入首页热门空间/热门内容接口
   - 热门部落区当前是写死数据

3. 搜索框接入真实跳转逻辑
   - 现在只有 UI，没有搜索动作

#### 中优先级

4. “了解更多”按钮补真实行为
   - 跳产品介绍页 / 滚动到下方内容 / 打开文档页

5. 游客点需要登录的导航时给更明确提示
   - 现在直接被路由守卫跳登录，体验上稍显生硬

### 结论

- 首页属于“展示层完成、数据层未接入”

---

## 3.2 个人中心 `Profile.vue`

### 当前状态

- 页面视觉丰富
- 但基本是 mock 数据驱动
- 没有形成“我的资料管理中心”的真实闭环

### 当前缺口

#### 高优先级

1. 接入真实个人资料加载
   - `getMyProfile()`

2. 接入资料编辑与保存
   - `updateProfile()`

3. 让“编辑资料”按钮具备真实动作
   - 弹编辑表单 / 跳编辑页 / 行内编辑

4. 让关注、粉丝、积分等数据改为真实接口返回

#### 中优先级

5. 补“免打扰设置”接入
   - `getMuteSettings()`
   - `updateMuteSettings()`

6. 补“标签订阅”接入
   - `getTagSubscriptions()`
   - `updateTagSubscriptions()`

7. Tab 内容改成真实分发
   - 动态
   - 帖子
   - 回复
   - 收藏
   - 打卡
   - 成就

#### 低优先级

8. 打卡日历与成就卡片接真实数据

### 结论

- `Profile.vue` 是当前最典型的“设计稿已成型，但业务仍未落地”的页面之一

## 3.4 广场 `Square.vue`

   - 公告

5. 空间 tabs 需要真正拆分为业务分区

   - 文件
   - 成员
   - 打卡
   - 设置

6. 补空间内发帖链路

### 结论

- 这是当前前台最值得重点补完的页面之一
- 建议把它当成一个独立迭代处理

## 4. 结构性问题清单

## 4.1 API 封装不统一

### 表现

- 大部分页面走 `src/api/*.ts`
- `AdminSensitiveWords.vue` 直接使用 `request()`

### 影响

- 不利于后续维护
- 不利于类型统一

### 建议优先级

中

---

## 4.2 类型定义滞后于真实接口使用

### 表现

- 如 `getPosts()` 的实际页面使用能力比显式类型更丰富

### 影响

- TS 提示可能不准确
- 增加重构风险

### 建议优先级

高

---

## 4.3 页面错误态很多是静默吞掉

### 表现

- 多个页面 catch 后只保留现状，不提示用户

### 影响

- 用户会看到空白或旧数据，不知道是否出错

### 建议优先级

中

---

## 4.4 页面视觉风格和业务成熟度不一致

### 表现

- 有些页面视觉很完整，但数据和交互很弱
- 如 `Profile.vue`、`SpaceDetail.vue`、`AiAssistant.vue`

### 影响

- 用户会误以为功能完整，实际操作时发现无效果

### 建议优先级

高

---

## 5. 建议的优先级排序

## P0：最优先补齐

1. 搜索页自动读取 `q` 并自动搜索

3. `AiAssistant.vue` 接真实 AI API
4. `Profile.vue` 接真实个人资料闭环

---

1. 首页数据真实化

3. 后台仪表盘接真实数据
4. 通知/消息页体验增强
5. 评论/记录分页优化

---

## P2：体验与一致性优化

1. 后台各表格页样式统一
2. API 模块封装统一
3. 错误态与空态统一
4. 个人中心 tabs 真正业务化
5. 更多 AI 增强能力接入

## 7. 总结

如果一句话概括当前前端现状：

**主业务骨架已经搭起来了，但仍有少数关键页面停留在“UI 已经很像成品、业务却还没完全落地”的阶段。**

最值得优先处理的不是再做新页面，而是把下面几块补实：

- 搜索联动闭环
- 空间详情闭环
- 个人中心闭环
- AI 页面闭环
- 消息实时性
- API 类型与封装统一

只要这几块补上，当前项目的前端完成度会明显从“可演示”提升到“更接近真实可用产品”。

# 高校轻量化学习社群平台 · 需求分析文档

> **项目代号**：CampusForum  
> **完整定名**：基于多租户开源架构与 AI 增强的高校轻量化学习社群平台  
> **试点单位**：桂林电子科技大学北海校区  
> **文档版本**：v1.0（2026-04）  
> **文档目的**：明确平台的目标用户、业务场景、功能与非功能需求，为后续技术设计、开发实施与项目验收提供依据。

---

## 1. 项目背景

### 1.1 行业背景

教育部《教育信息化 2.0 行动计划》《教育数字化战略行动》以及《关于深入推进高等学校数字校园建设的实施意见》明确提出：高校应建设"以学生为中心、以学习为本"的数字化校园生态，鼓励高校自主、可控、低成本地开展信息化建设。当前高校学生在线交流主要依赖三类工具：

1. **通用社交软件**（QQ 群、微信群、钉钉群等）：信息流式碎片化、检索差、与学校无组织绑定、数据归属第三方。
2. **集中式商业校园社区**（如某 App、某校园圈）：闭源、收费、数据上云、无法私有化、各校无法定制。
3. **学院或个人自建论坛**（Discuz、phpBB、定制开发）：技术老旧、维护成本高、UI 体验差、无统一开源生态。

三者均无法同时满足"**专业圈层私密、数据归属学校、低成本部署、可二次开发、统一开源生态**"这一组核心诉求。

### 1.2 用户痛点（基于实地访谈与调研归纳）

| 编号 | 痛点描述 | 涉及人群 |
|---|---|---|
| P1 | QQ/微信群消息流式刷屏，重要学习资源容易被覆盖、难以检索 | 学生 |
| P2 | 校内缺少按专业/班级/社团划分的私密学习空间，专业讨论混入生活信息 | 学生、社团 |
| P3 | 资料和历年题在多个群聊中重复转发、版本混乱、无沉淀机制 | 学生 |
| P4 | 自律打卡多依赖个人小程序，缺少基于校园/专业熟人圈的同伴激励 | 学生 |
| P5 | 学院/学校缺少自有数据可控、可定制的学生交流平台 | 学校管理方 |
| P6 | 现有商业方案部署门槛高、成本高、各校难以差异化定制 | 学校信息化部门 |
| P7 | 现有平台缺少 AI 辅助内容理解能力，话题信息密度高时阅读成本大 | 学生 |

### 1.3 项目意义

- **学生侧**：提供专业、私密、有同伴感的学习与生活交流空间，沉淀学习资源、激励自律行为。
- **学校侧**：以零成本获得自主可控、数据归属本校、可定制扩展的官方学习社群平台。
- **行业侧**：以 MIT 协议开源，构建国内首个面向高校的"轻量化、多租户、AI 增强"开源校园社区生态，为高校信息化建设提供可复用样本。

## 2. 项目目标

### 2.1 总目标

研发一款**面向全国高校、以学习互助为核心、采用"全校广场 + 专业学习空间"双层架构**的开源轻量化校园学习社群平台，支持单实例多租户部署与一键本地私有化部署，融合 AI 内容辅助能力，并以桂林电子科技大学北海校区为首个试点单位完成校内试运行。

### 2.2 分目标

1. **业务目标**：完成 8 大功能模块（用户/广场/专业空间/学习互助/打卡/资源/通知/管理后台）的设计与开发。
2. **技术目标**：完成多租户架构、数据隔离机制、Docker 一键部署体系、AI 服务接入设计与实现。
3. **生态目标**：项目源码、部署脚本、开发文档、用户手册以 MIT 协议在 GitHub 持续开源维护。
4. **研究目标**：形成关于"高校开源多租户社群平台架构与隔离机制"的研究报告 1 篇与学术论文 1 篇。
5. **落地目标**：在桂电北海校区面向不少于 3 个学院/专业开展试运行，覆盖不少于 500 名活跃用户，收集试运行报告。

### 2.3 成功标准（KPI）

| 维度 | 指标 | 目标值 |
|---|---|---|
| 功能完成度 | 8 大模块功能验收通过率 | ≥ 95% |
| 性能 | API P95 响应时间 | < 500 ms |
| 性能 | 首屏首次有效绘制（FCP） | < 1.5 s |
| 部署体验 | 私有化部署一键脚本执行成功率 | ≥ 95% |
| 试运行 | 校内活跃用户数（DAU） | ≥ 100 |
| 试运行 | 累计注册用户数 | ≥ 500 |
| 开源生态 | GitHub Star 数（试运行结束时） | ≥ 50 |
| 学术成果 | 软著 + 论文 + 研究报告 | 各 1 项 |

## 4. 用户画像与典型场景

### 4.1 用户画像

#### 画像 A：学习驱动型大二学生（典型代表 70%）

- 网络工程专业大二，每天在线时长 4 小时左右
- 痛点：群消息太多看不过来、找历年题靠学长学姐
- 期望：能在专业圈高效问答、有人监督打卡、资料分类沉淀

#### 画像 B：社团/学生组织负责人（典型代表 15%）

- 计算机协会会长，需要管理 数十位 成员
- 痛点：QQ 群里通知容易被刷掉、活动报名靠手工统计
- 期望：能创建独立空间发布通知、置顶活动、统计成员动态

#### 画像 C：学校管理员（典型代表 5%）

- 学院学工办老师 / 信息化中心管理员
- 痛点：QQ 群信息无法监控、舆情风险大、数据不可控
- 期望：可以查看全平台数据、违规内容快速处理、本校数据完全独立

#### 画像 D：开源贡献者（典型代表 10%）

- 高年级学生 / 其他高校开发者
- 痛点：现有开源校园系统老旧、文档不全
- 期望：基于本平台二次开发、本校私有化部署

### 4.2 典型业务场景（User Journey）

#### 场景 1：新生入校加入专业学习空间

> 大一新生小李通过校园门户跳转到平台，使用学号统一认证登录 → 首页提示完善个人信息 → 系统根据学号自动推荐"网络工程 22 级"专业空间 → 申请加入并由空间管理员审核通过 → 进入专业空间看到学长学姐沉淀的历年题、入门书单与答疑帖。

#### 场景 2：学习互助问答闭环

> 同学小王在数据结构作业中遇到红黑树旋转问题 → 在专业空间发起带"求助"标签的提问帖 + 悬赏 20 积分 → AI 自动生成问题摘要并打上技术标签 → 同空间的学长 30 分钟内作答 → 提问者采纳并自动支付积分 → 该帖沉淀进"专业知识库"被后续同学搜索复用。

#### 场景 3：自律打卡同伴激励

> 大二同学小张参与"21 天英语打卡"挑战 → 每日上传打卡内容 → AI 自动检测打卡内容真实性（如非空、含图片或文字） → 系统记录连续天数并展示在专业空间排行榜 → 连续 21 天后获得"自律者"成就徽章并出现在个人主页。

#### 场景 4：空间管理员管理动态

> 计算机协会会长作为空间管理员，发布招新通知并置顶 → 收到 300+ 加入申请 → 在管理面板批量审核 → 设置敏感词与发帖规范 → 一键导出本空间成员名册用于线下活动签到。

#### 场景 5：学校私有化部署

> 某高校信息化中心希望部署本平台 → 在 GitHub 下载发布包 → 修改 `.env` 配置（域名、数据库密码、AI Key 可选） → `docker compose up -d` 一键启动 → 5 分钟内完成本校独立站点部署，数据完全隔离。

#### 场景 6：AI 辅助内容浏览

> 用户进入热门话题帖（楼层 200+） → 点击"AI 摘要"按钮 → AI 在 3 秒内生成本帖核心观点 + 主要分歧 + 关键回复定位 → 用户快速把握话题脉络，不必从头翻阅。

## 5. 功能性需求

## 6. 非功能性需求

### 6.1 性能需求

| 指标 | 要求 |
|---|---|
| API 平均响应时间 | < 200 ms |
| API P95 响应时间 | < 500 ms |
| 首屏首次有效绘制（FCP） | < 1.5 s |
| 单节点支持并发用户 | ≥ 1000 |
| 数据库单表预期最大记录数 | 1000 万 |
| AI 摘要响应时间 | < 5 s（接 OpenAI 协议） |

### 6.2 安全需求

- 密码采用 BCrypt 加盐哈希存储
- 所有接口基于 JWT + Redis 黑名单的鉴权机制
- 防 SQL 注入（MyBatis-Plus 参数化查询）、防 XSS（前端 v-html 白名单 + 后端 HTML 过滤）
- 防 CSRF（同源策略 + Token 校验）
- 文件上传白名单 + 文件头校验 + 病毒扫描预留接口
- 接口限流（基于 Redis + Lua）
- 操作审计日志全留存（管理员操作）
- 多租户模式下租户间数据零泄漏（必须经过隔离测试）

### 6.3 可用性需求

- 移动端优先响应式设计，覆盖 360px / 768px / 1280px 三档
- 关键功能 3 秒内可达（不超过 3 次点击）
- 支持深色模式
- 关键页面具备无障碍可访问能力（aria-label）
- 支持 PWA 安装、桌面图标、离线访问历史浏览

### 6.4 可维护性 / 可扩展性需求

- 前后端分离，前端可独立部署到 CDN
- 后端模块化分层，关键能力（搜索、AI、存储）以接口形式抽象，便于替换实现
- 完整 OpenAPI 3.0 文档（基于 SpringDoc）
- 单元测试覆盖率 ≥ 60%（核心模块 ≥ 80%）
- 提供完整开发文档、部署文档、API 文档、二次开发指南

### 6.5 兼容性需求

- 浏览器：Chrome 90+ / Edge 90+ / Safari 14+ / 移动端主流浏览器
- 数据库：MySQL 8.0+，预留 PostgreSQL 适配空间
- 部署：Linux（Ubuntu 20.04+ / CentOS 7+）、支持 Docker 20.10+

### 6.6 开源与合规需求

- 项目采用 MIT 协议开源
- 所有第三方依赖须协议兼容（无 GPL 强传染依赖）
- 提供 `LICENSE`、`CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、`SECURITY.md`
- 隐私合规：符合《个人信息保护法》，提供用户数据导出与注销能力
